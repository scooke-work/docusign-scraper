# Define Workflow Participants

Learn how to define, configure, and manage participant roles within your workflow steps to control how users interact with your agreement automation.

When you create a workflow, consider the roles that will participate in your agreement process
and any data that must be provided at the beginning of the process. Who are the
individuals and entities that will provide information, sign documents, and participate
in any other way? In a workflow, these are called "participants".

Most workflow steps require you to specify a participant. The step participant is the role that
will take some action to complete the workflow step. Certain steps require
specific participant information. For example, a participant assigned to a Send Documents for Signature step requires variables (or hardcoded values) for their name and email
address. You define this information in the workflow step by mapping variables from previous workflow steps.

## Add participants to a workflow

Participants are created for each workflow. When you create a new workflow, you can
configure all of your participants up front. Or you can configure participants as
you build your workflow and add steps.

When you create a workflow step that requires a participant assignment, you can
select from your list of configured participants or add a new one.

As you add participants, give them meaningful role names that describe the kind of
activities and responsibilities they will have in your workflow.

## How participants enter a workflow

Assigning a participant to a step grants them access to that step and defines at what point
they can enter the workflow. Only participants assigned to a particular step will be
able to see and complete the steps.

There are two ways that a participant can enter a workflow:

- **Participants that enter the workflow directly.** For From a Link, Manually in Docusign, and API Call (human start) methods, the user who launches the
  process is directed to the first workflow step and assigned as the
  first participant by default. See the article [Workflow Start Methods](https://support.docusign.com/s/document-item?bundleId=yff1696971835267&topicId=ztb1727892686033.html&_LANG=enus "Review the different automated and manual trigger methods available to initiate processes in Workflow Builder, including links, events, API calls, and CLM integrations.").
- **Participants are invited to the workflow.** Any other participants who need
  to enter the workflow must be invited. This invitation provides a URL specific
  to the steps that are assigned to them. For example, for a Link start type
  workflow, the person using the link enters the workflow directly. The second
  participant of the workflow is invited to the workflow so that they can access
  their steps. Or for the API Automated Process start, any participants that are
  part of the workflow, including the first workflow step, need to be invited so
  that they can access their workflow steps. See the topic [Configure a Set Up Invite Step](https://support.docusign.com/s/document-item?bundleId=yff1696971835267&topicId=kfx1698167344720.html&_LANG=enus "Learn how to use a Set Up Invite step in your agreement workflow to invite participants to join the workflow.").

## Define a participant's information

Some steps like Send Documents for Signature, Verify Someone’s Identity, and Set Up Invite require
participants' information like Name, Email, and Phone. This information could be
coming from multiple sources.

Here are some examples of using the different workflow start methods:

- For the From a Link start method, the person that starts the workflow can provide their own
  information in a Web Form step.
- For the Manually in Docusign start method, the person who launches the process can provide
  participant information in a Collect Data with Web Forms step.
- For API and CLM start types, participant information can come from an external system
  as a payload. This information is represented as starting variables that can be used as
  placeholders to map to the participant information in subsequent workflow steps.

Once you add a step to your workflow that requires participant information, you provide the data directly in the step configuration. You can map
variables from prior steps or hardcode specific values. ![](https://docusign-be-prod.zoominsoftware.io/api/bundle/yff1696971835267/page/images/specify-participant-information.png?token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJjdXN0b21lciI6ImRvY3VzaWduX3Byb2R1Y3Rpb24iLCJleHAiOjE3ODE4MDU2ODYsInNoZWFmIjoieWZmMTY5Njk3MTgzNTI2NyJ9.f4wlRCLjPUbBWMhyJ9FHk6GU1XNcmKDnmLrBlTaWHhw&_LANG=enus)

Once defined, participant information is
set at a global level for the workflow. Any steps that come later in the workflow
use the information defined in the step.

If you need to change a participant's information at some point in your workflow,
you can map new values in any step that uses the information. The revised participant information is used for all subsequent steps, unless revised again in another step.

Note: If your workflow generates an identity claim from a Verify
Someone's Identity step, this claim can be applied in a
Send for Signature step of the same workflow instance.
However, if you update or change the participant's name after the identity claim is
established, then it is no longer available to subsequent workflow steps.
