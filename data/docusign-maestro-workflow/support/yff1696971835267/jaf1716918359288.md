# Configuration Details for Complex Branching Rules

Read about nesting IF statements and using advanced operators in a branching rule workflow step.

This topic provides information on creating more complex branching rules and provides some illustrative examples.

## Nested IF statements

You can design a workflow with multiple nested branching rules. You construct nested IFs by adding branching rules to the outcomes of another branching rule. Nested IFs support more complex conditional logic than just a single rule that provides a binary, TRUE/FALSE outcome. You can nest up to eight levels of rules.

If you delete a branching rule, you select whether to keep the True or False path steps in your workflow. You can also choose to delete all steps from both paths.

## Multiple AND or OR conditions

You can configure a single branching rule with up to five AND or OR conditions. You can mix AND and OR conditions to build multiple groups of conditions.

- A rule with multiple AND conditions evaluates to true if the conditions are all true, and false if any conditions are false.
- A rule with multiple OR conditions evaluates to true if any of the conditions are true, and false if all conditions are false.
- A rule with a mix of AND and OR conditions evaluates to true if any group of conditions evaluates to true, and false if all groups are false.

## Advanced conditionality

Branching rules support advanced conditionality logic. The available conditions depend on the data type you are using for your rule.

Booleans
:   For variables that are boolean and have only two possible values, the branching rule operators are `Is` and `Is not`. The results are `True` or `False`.
:   Examples:

    - Checkbox field is checked or unchecked.
    - Radio button is selected or unselected.
    - Identity verification check passed or failed.

Strings
:   For supported field types (text, note, email, name) that can contain string values, the standard comparison operators are provided. These can evaluate if the variable value equals, contains, starts with, or ends with a specified result.
:   Examples:

    - Text field contains the value `California`.
    - Email field ends with `.org`.
    - Text field does not equal the value `New Jersey`.![Screenshot of the rule operators for a string comparison.](https://docusign-be-prod.zoominsoftware.io/api/bundle/yff1696971835267/page/images/advancedruleoperators.png?token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJjdXN0b21lciI6ImRvY3VzaWduX3Byb2R1Y3Rpb24iLCJleHAiOjE3ODE4MDU4MzQsInNoZWFmIjoieWZmMTY5Njk3MTgzNTI2NyJ9.5lqtNMJHcOV58i04qkKTeAqk0locHsUib65JNU9r0AU&_LANG=enus "Screenshot of the rule operators for a string comparison.")

Dropdowns
:   For dropdown field types from a web form or eSignature step, standard comparison operators are provided to evaluate if the variable value equals a specified dropdown value. To select the result the rule will look for, the rule configuration presents the list of dropdown values from the mapped field to select from. ![Screenshot of configuration for a rule checking the value of a dropdown field from a web form step.](https://docusign-be-prod.zoominsoftware.io/api/bundle/yff1696971835267/page/images/dropdown_rule.png?token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJjdXN0b21lciI6ImRvY3VzaWduX3Byb2R1Y3Rpb24iLCJleHAiOjE3ODE4MDU4MzQsInNoZWFmIjoieWZmMTY5Njk3MTgzNTI2NyJ9.5lqtNMJHcOV58i04qkKTeAqk0locHsUib65JNU9r0AU&_LANG=enus "Screenshot of configuration for a rule checking the value of a dropdown field from a web form step.")

Dates
:   For date fields originating from a web form, eSignature step, instance date, or a starting variable, comparisons of equality, inequality, and dynamic comparison are supported. For web form date fields, use the (Normalized) version of the field. For other variables, you can select any variable of type = date for a comparison.![Screenshot of a rule evaluating a date field and the list of available operators.](https://docusign-be-prod.zoominsoftware.io/api/bundle/yff1696971835267/page/images/conditional_rule_date_operators.png?token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJjdXN0b21lciI6ImRvY3VzaWduX3Byb2R1Y3Rpb24iLCJleHAiOjE3ODE4MDU4MzQsInNoZWFmIjoieWZmMTY5Njk3MTgzNTI2NyJ9.5lqtNMJHcOV58i04qkKTeAqk0locHsUib65JNU9r0AU&_LANG=enus "Screenshot of a rule evaluating a date field and the list of available operators.")

Numbers
:   For number fields originating from a web form or a starting variable, the standard numerical comparisons are supported. For web form number fields, use the (Normalized) version of the field. For starting variables, you can select any variable of type = numbers for a numerical comparison. ![Screenshot of the configuration of a branching rule for a numerical comparison.](https://docusign-be-prod.zoominsoftware.io/api/bundle/yff1696971835267/page/images/branchingrulenumericalcomparison.png?token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJjdXN0b21lciI6ImRvY3VzaWduX3Byb2R1Y3Rpb24iLCJleHAiOjE3ODE4MDU4MzQsInNoZWFmIjoieWZmMTY5Njk3MTgzNTI2NyJ9.5lqtNMJHcOV58i04qkKTeAqk0locHsUib65JNU9r0AU&_LANG=enus "Screenshot of the configuration of a branching rule for a numerical comparison.")

Currency
:   Currency fields can originate from starting variables or from data gathered through steps in your workflow, such as a web form or eSignature envelope. When mapping a currency field, there are three versions of the field. You can use any one of them in a branching rule.

    - **Currency field name**: contains the currency code and numeric amount
    - **Currency field name (Double Value)**: contains the numeric amount
    - **Currency field name (Currency Code)**: contains the currency code

      Note: If your branching rule evaluates the full currency field rather than its individual components (`Double Value` or `Currency Code`), the rule requires all parts of the currency data to match your criteria. All conditions must evaluate to true for the overall outcome to be true.

      ![Screenshot of a branching rule configured to check if TotalContractValue is greater than or equal to $1,000.00 USD.](https://docusign-be-prod.zoominsoftware.io/api/bundle/yff1696971835267/page/images/currency_branching_rule_ex.png?token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJjdXN0b21lciI6ImRvY3VzaWduX3Byb2R1Y3Rpb24iLCJleHAiOjE3ODE4MDU4MzQsInNoZWFmIjoieWZmMTY5Njk3MTgzNTI2NyJ9.5lqtNMJHcOV58i04qkKTeAqk0locHsUib65JNU9r0AU&_LANG=enus "Screenshot of a branching rule configured to check if TotalContractValue is greater than or equal to $1,000.00 USD.")

## Merging branches

You can configure a branching rule to merge the True and False branches. This is optional and useful if you want to combine the paths to use a single step outcome. You can add steps to each path to customize the workflow based on the initial outcome of the rule.

Note: You can also merge a multi branch rule. A multi branch rule can have up to 15 branches. You can merge all branches, or any two or more branches.

Important things to consider when opting to merge a branching rule:

- Variables and files generated from steps inside of a merged branching rule will not be available to use in steps occurring after the merge. Do not merge a rule if you are gathering data or generating files you want to use later in the workflow.
- You can nest another branching rule within the merged paths. The paths of a nested branching rule will also merge into a single outcome.
- You can edit the step to remove the merge setting. Any steps in the rule occurring before the merge point are retained on the branch where they exist. For any steps after the merge, you can delete them or select which path to move them to.
