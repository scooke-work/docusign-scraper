# Test a Web Request Configuration

Confirms that a configured Make a Web Request step successfully communicates with an external system and returns the expected data.

Use this procedure after you configure a Make a Web Request step to confirm that the step can successfully call the external system and return the data you expect.

## Before you begin

Before you begin, make sure:

- You have a published workflow with the web request step. The step is configured with a connection or endpoint that is appropriate for testing. For example, a sandbox connection.
- You understand what a successful response looks like (for example, a specific status code or field value in the response).

1. Start a test run of the workflow.
   1. From Workflow Builder, start a new instance using test data that exercises the Web Request step.
   2. If your workflow collects participant input, enter realistic but non-production values.
2. Monitor the Web Request step during the run.
   1. Open the workflow run details.
   2. Locate the Web Request step in the run timeline.
   3. Check whether the step shows a successful status or an error.
3. If the step succeeds, verify the response in Workflow Builder.
   1. Confirm that the response variables you mapped in the Web Request step are populated.
   2. Check that any downstream steps that use those variables behave as expected (for example, the workflow follows the correct branch based on a risk score or status value).
   3. If needed, run an additional test with different inputs to confirm that the workflow reacts correctly to different responses.
4. If the step fails, review the error and make basic corrections.
   1. Read any error message or status code shown for the Web Request step.
   2. Check for common configuration issues:

      - The URL is misspelled or missing required path segments.
      - The connection is pointing to the wrong environment (for example, production instead of sandbox).
      - Required headers or body fields are missing or use the wrong data type.
   3. Update the Web Request step configuration as needed, then save the workflow and run another test.
5. (Optional) Confirm the external system was updated correctly.
   1. If the Web Request is supposed to create or update data in another system, sign in to that system or use its logs to confirm that the call arrived and was processed as expected.
   2. If the external system does not show the expected change, share the workflow run details and any error messages with your technical team so they can check the API configuration.

When you are satisfied with the test results:

- Keep the test configuration if it matches your production requirements, or switch the connection or endpoint to the production values your technical team has approved.
- Save the workflow and follow your normal process for enabling or publishing it for broader use.
