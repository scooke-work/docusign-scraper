# General Limitations and Considerations for Workflow Builder

Review important account limits, supported regions, and functional considerations before building automated processes in Workflow Builder.

Workflow Builder is a new platform to help you build and deploy customized workflows that automate and accelerate your agreement processes without writing any code. As you explore this exciting new offering, there are some general limitations and other considerations you should be aware of. Docusign will continue to develop the platform, and potentially address some of these limitations in future releases.

In addition to these general limitations, if you use a workflow step to get signatures you should review the topic [Template Requirements](https://support.docusign.com/s/document-item?bundleId=yff1696971835267&topicId=xnw1709852160602.html&_LANG=enus "Important information for process builders to consider before starting to develop workflows that include a Prepare eSignature Template step. When you prepare a template to use in a Send Documents for Signature step, there are several requirements and limitations that must be respected to configure the step successfully.").

## General limitations

Supported regions and languages
:   Workflow Builder is available globally in all supported sending languages.

FedRAMP and GovRAMP certification compliance
:   For customers in regulated industries, Workflow Builder and its pre-built templates are FedRAMP and GovRAMP certified. However, for these account types, the following Workflow Builder features are not supported:

    - Verify Someone's Identity step
    - Collect Data from an ID step
    - SMS and WhatsApp delivery methods for notifications

Number of workflow definitions
:   Your organization's account plan determines how many published and paused workflows you can have at a time. Once your account reaches this limit, you must unpublish a workflow before you can publish a new one. There is no limit on the number of draft workflows that you can have.

Standalone web forms
:   Workflow data can be gathered using web forms. Use a standalone web form for this action. Web forms created from templates are not supported. See the information on standalone web forms in the topic [Web Form Requirements and Data-Mapping Information](https://support.docusign.com/s/document-item?bundleId=yff1696971835267&topicId=oha1716325069710.html&_LANG=enus "Review the configuration rules, standalone formatting options, and precise field-matching requirements needed to map data between web forms and agreement templates.") for more information.

Data types and character limits for variables
:   For workflows with start methods designed to initiate the workflow from an external system, you can specify variables. These variables receive payload data from the external system. For example, your customer starts a workflow from your website to submit an application. They've logged in to their account, so when they start the application, the system passes their name, contact information, and account number into the process.

    There are several different types of data that you can define for a starting variable. Each of these supports specific input values. For details on the supported data types, see the help topic [Data Types for Workflow Variables](https://support.docusign.com/s/document-item?bundleId=yff1696971835267&topicId=ogx1738026109020.html&_LANG=enus "Look up details about the data types that you can use to pass data to workflow variables.").

Instance names
:   You can specify unique workflow instance names by using workflow variables. Instance names can be defined in the publishing configuration for a workflow. For workflows that use an API start method, the API payload can pass the `instanceName` value. The `instanceName` value takes precedence over the instance name defined in the workflow publishing configuration.
:   The following variable types are supported for constructing an instance name:
:   - Strings
    - Numbers
    - Dates
    - Email
    - Phone
    - Boolean

Direct signing account prerequisites and 21 CFR Part 11 accounts
:   Direct signing is an option for the first signer in a Send Documents for Signature step to go directly from a prior workflow step (such as a web form or ID Verification step) into the signing session. To configure a Send Documents for Signature step for direct signing, your Docusign account must have the **Enable In Session Signing** setting turned on. If this feature is not turned on for your account, workflows using direct signing will fail at that step with the following error:

    ```
    "This Account lacks sufficient permissions. In-Session permission required when specifying a captive recipient."
    ```

    If you need access to this capability, contact Docusign Support or your account team for assistance.

    Direct signing is not available for Docusign IAM accounts with a Life Sciences Part 11 module.

Workflow sizing and resource limits
:   Workflows do not have a hard step maximum, but rather an internal size limit based on total underlying system actions. Standard workflows comfortably support 30 to 40 steps, but resource-intensive actions—such as Send Documents for Signature or Collect Data with Web Forms—consume more platform capacity and may lower your total allowed step count. If a workflow design exceeds these internal resource thresholds, a size warning or error will appear on the canvas at publish time to prevent deployment.
