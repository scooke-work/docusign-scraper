# Access and Permissions for Extension App Connections

Outlines the access levels and permissions required to edit or publish workflow steps that use private or shared extension application connections.

Docusign and Workflow administrators can manage all workflows on an account. However, if a workflow includes an extension app step that uses a private connection, specific **connection guardrails** apply. These guardrails protect the private connection data while allowing the administrator to still manage and publish the overall workflow. Configurator details for private connections remain locked and hidden from users who don't own the connection — protecting sensitive resources.

While the administrator cannot view or edit the locked configuration of a private connection that they do not own, they maintain full oversight of the process. This topic explores these access levels and permissions for workflows that include third-party (3P) extension app connections in more detail.

## Editing workflow steps

Workflow steps for extension applications can be configured to use either a private or shared connection to a third-party application. The connection type affects who can configure the step.

Shared connection
:   Users with the **Workflows: Manage** permission can edit the step configuration for the extension application.

Private connection
:   Only the connection owner can edit the step configuration. For all other users, including those with the **Workflows: Manage** permission, the step will display in a locked or restricted state. These users cannot view or edit the step configuration or make any changes to the connection.

    These users can still:

    - Edit other steps in the workflow (for example, Web forms, confirmation screens).
    - Delete the extension application step.
    - Contact the private 3P app connection owner to have them make changes to the step.
    - Run the workflow.

## Collaborative publishing

Users with the **Workflows: Manage** permission can publish a workflow even if it contains steps with private connections they do not own. Multiple users can contribute to and publish a single workflow, even if it contains steps tied to different private third-party accounts (for example, Salesforce or Google Drive).

- When a connection owner adds a private 3P account to a workflow step, they **authorize** the workflow to use their connection credentials. This authorization remains valid even if another user publishes the workflow.
- While any workflow manager can publish the workflow, the private configuration details (such as API keys or login credentials) remain locked and hidden from everyone except the original connection owner.
- No single user must own all connections in a workflow to publish it.

## Running workflows

Any user with the **Workflows Manage**, **Create**, or **Prepare** permission can start an instance of a published workflow, even if it includes steps with private or shared connections to third-party extension apps. The connection type does not restrict the ability to trigger a workflow.
