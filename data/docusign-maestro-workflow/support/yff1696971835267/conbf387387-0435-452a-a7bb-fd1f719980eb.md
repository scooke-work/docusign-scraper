# Connect to Any Service Using Web Requests

Docusign Workflow Builder allows you to trigger external APIs using the Make a Web Request utility step.

The Make a Web Request step lets you send HTTP requests (GET, POST, or PUT) to almost any external or internal service as part of a Workflow Builder process, without waiting for a dedicated connector or app. This makes it easier to connect your agreements to systems such as CRMs, procurement tools, risk-scoring services, and internal APIs that are not yet available as first-class integrations.

## What the Web Request step does

When you add a Make a Web Request step to a workflow, Workflow Builder can:

- Call an HTTP endpoint that you specify, using a supported method (**GET**, **POST**, or **PUT**).
- Pass data from earlier steps (such as participant inputs, agreement metadata, or values returned by other steps) into the request URL, headers, or body.
- Capture the response for GET requests and expose it as new variables you can reference in later steps, including for routing and branching decisions.

You can think of the Web Request step as a universal connector: Technical teams configure the underlying HTTP behavior. Process builders simply choose the step, select the right connection, and map which workflow data should be sent and received.

## Typical use cases

Use a Web Request step when you need Workflow Builder to talk to another system in the middle of a process. Examples include:

- Looking up data in a system of record and using that data to drive downstream steps. For example, pulling customer, pricing, or risk data from a CRM or internal API.
- Updating an external system when an agreement reaches a certain point. For example, pushing status updates into ServiceNow, Coupa, SAP Ariba, or another workflow tool.
- Calling Docusign CLM or other Docusign APIs from within a workflow, so you can orchestrate agreements alongside existing CLM processes.

Because the step is generic, the same configuration pattern can support many different APIs and business processes over time.

## How it fits into Workflow Builder

The Web Request step is a first-party processing step in [Workflow Builder](https://support.docusign.com/s/document-item?bundleId=yff1696971835267&topicId=uhm1697826856678.html&_LANG=enus "Learn the basics of how to create an agreement workflow. Process creators and account administrators can create workflows to automate the agreement process."), alongside steps that collect data, prepare documents, and send envelopes. Once enabled for your account, it is listed in the Utility section of the step selector. You can add it to any eligible workflow without installing a separate app.

From a process builder’s perspective, using the step typically involves three high-level decisions:

1. Choose which Web Request connection to use. For example, a connection that your administrator set up for a specific API.
2. Specify what the step should send. This includes the HTTP method, URL pattern, and which workflow variables to include in the headers or body.
3. Decide how to use the response. For example, mapping returned fields into workflow variables, or driving branches based on the status or content of the response.

## Availability and prerequisites

The HTTP Web Request Extension is a premium capability and is available to customers on qualifying Docusign Intelligent Agreement Management (IAM) plans such as IAM Professional and above. The HTTP app is auto-installed for all authorized accounts and does not require manual installation from the [Docusign App Center](https://support.docusign.com/s/document-item?bundleId=ous1698169987748&topicId=ijy1698170251179.html).

Before process builders use the step, an administrator or technical owner should configure at least one Web Request connection. This step currently supports Basic Auth (user name/password) connections. The connection defines the credentials to use for the API the step will call. At runtime, the Web Request step uses the credentials you selected and injects them into the request header.

Important: Connection credentials are securely stored and not exposed in workflows using them in a Web Request step. The workflow contains only an ID that is used to retrieve the credentials at runtime.
