# Configure a Make a Web Request Step

Use a Make a Web Request step to call an external or internal service from Workflow Builder to look up data or push an update without a dedicated app.

Use a Make a Web Request step when you want Workflow Builder to call another system in the middle of a process (for example, to look up data or push an update). Follow these steps to configure the Web Request step.

## Before you begin

Confirm that you have access to the HTTP Web Request feature:

- Your account includes access to the [HTTP web request extension app](https://support.docusign.com/s/document-item?bundleId=ous1698169987748&topicId=ijy1698170251179.html). This app is automatically installed for Docusign IAM premium accounts.
- An administrator or technical owner has created at least one Web Request connection for the target API. Currently, Basic Auth connections are supported (username and password). The connection provides valid credentials to authenticate access to the target API.
- You know which endpoint, method, and data your technical team expects you to use.

1. Open your workflow in Workflow Builder.
2. On the canvas, select the plus icon (`+`) where you want to add the step.

   Add the step where the external call should occur (for example, after collecting participant data and before sending an envelope).
3. In the step selector, choose the Make a Web Request step from the Developer section.

   The step configurator opens for you to define the request details.
4. Select the connection to use for the request.

   - If more than one connection is available, choose the option that matches the target system (for example, "Risk Scoring API – Production" or "CLM API – Sandbox").
   - If you do not see the connection that you need, contact your administrator to create or share it.

   Note: The connection controls how Workflow Builder authenticates with the external service. You do not need to enter credentials directly in the step.
5. Define the destination method and endpoint URL.
   1. Choose the HTTP method:

      - GET to retrieve data from the external system.
      - POST to send new data or trigger an action.
      - PUT to update existing data.
   2. Enter the endpoint URL that the web request will access.
6. Define the request by configuring a JSON payload and passing additional metadata about your request.

   If you selected POST or PUT and need to send a request body, configure the body of the request:

   1. Enter the JSON input provided by your technical team.
   2. Use the variable picker to insert workflow variables as values for a key/value pair into the payload so the request includes participant input or other step data.    3. Confirm that the data types (for example, numbers, dates, booleans) match what the API expects.
7. (Optional) Configure any additional headers that your API requires.

   Define custom HTTP headers as key-value pairs to provide necessary metadata required by the destination API. You can add a maximum of 10 headers per request.

   1. In the Headers area, add a row for each header.
   2. Enter the header key (name). Limit: 256 characters.
   3. Enter a static value (manual entry) or map a workflow variable into the value if the header should change per request. Limit: 4,000 characters.

      Important: The Authorization header is automatically populated at runtime with the values from the connection selected for the step.
8. (Optional) Configure query parameters.

   Use query parameters and key-value pairs to filter, sort, or search for specific data at the destination endpoint without needing to manually edit the base endpoint URL string. You can add a maximum of 10 query parameters per request.

   1. In the Query Parameters area, add a row for each parameter.
   2. Enter the parameter key (name). Limit: 256 characters.
   3. Enter the parameter value. You can enter a static text value (manual entry) or use the variable picker to insert workflow variables (such as an account ID or envelope ID) so the request is tailored dynamically to each workflow run. Limit: 4,000 characters.

      Note: The configurator automatically URL encodes the values that you enter. This ensures that any spaces or special characters passed by variables do not cause the HTTP request to fail.
9. Define the output variables for the response data.

   For GET requests, generate response variables or enter the response manually and map these to workflow variables. This way, later steps can use the data returned by the external system

   1. Select Run Request.

      The Code Editor opens with the request response. If the request couldn’t run, you can enter a sample response manually and select Regenerate to generate response variables.
   2. Map data types to variables.

      For each generated variable, review and confirm the data type selected is accurate. Edit the data type as needed.
   3. Select Save and Close.
10. Select Apply.

    Your Make a Web Request step is configured.
