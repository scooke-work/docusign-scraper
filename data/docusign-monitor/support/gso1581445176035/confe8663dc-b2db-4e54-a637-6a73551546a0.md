# Splunk Troubleshooting for Docusign Monitor Setup

Learn about technical resolutions for common setup and data ingestion errors that you may
encounter when configuring the Docusign Monitor Splunk Add-on
(TA).

Here are technical resolutions for common setup and data ingestion errors that you may encounter
when configuring the Docusign Monitor Splunk Add-on (TA).

## Version and compatibility Issues

Docusign Monitor supports the latest Splunk platforms.

Prior versions (9.2.x and earlier) with older add-ons may exhibit dependency and
library errors. Upgrade Splunk and the Docusign Monitor Add-on to
avoid recurring failures.

The Latest Add-on is available in [Splunkbase](https://splunkbase.splunk.com/).

**Resolution**

- Deploy the latest Docusign Monitor Add-on (TA)
  version to mitigate known Python JWT and cryptography dependency loading
  failures.
- Where feasible, upgrade the Splunk platform to **9.3+**.

Known errors include:

- `No module named '_cffi_backend'`
- `'str' object has no attribute 'decode'`
- Python JWT/cryptography dependency loading issues. Monitor license
  and organization entitlement

## Monitor license and organization entitlement

**Organization entitlement validation**

Confirm that Docusign Monitor is globally enabled at the
**Organization level**. Monitor is an organization-wide product, not a
single-account feature.

In Account Admin, verify that the client app has the required scopes and
reissue credentials if they are updated.

- `signature`
- `impersonation`
- `organization_read`

If the Admin dashboard shows Monitor, but Splunk widgets fail with a 403
error, it is usually a scope and entitlement mismatch: not a network outage.

**Propagation Delay**

Allow adequate time (typically a few hours) for entitlement caches to
propagate across Demo and Production environments after the initial activation.

## Required OAuth scopes for the client application

**Mandatory Scopes**

The client application that Splunk uses must possess, at a minimum, these
OAuth scopes:

- `signature`
- `impersonation`
- `organization_read`

**403 Errors**

A 403 Forbidden status on Monitor data inputs is the most frequent symptom
of insufficient scopes or entitlement, even if token authentication succeeds.

## Data input configuration and endpoints

**Input endpoints problem**

The Splunk Add-on requires three distinct data inputs with unique external
endpoints:

- `Account Users`
- `Organization Accounts`
- `Org Events`

**Diagnosis**

If ingestion fails for a single input while others succeed, the root cause
is typically an endpoint misconfiguration. It could also be a missing scope for that
specific resource.

## RSA key format and parsing issues

**Key format error**

If this error is returned.

`Could not parse the provided public key`

Verify that the key is encoded as PKCS#8 in the standard Privacy-Enhanced
Mail (PEM) delimiter:

`-----BEGIN PUBLIC KEY----- … -----`

`END PUBLIC KEY-----`

**Resolution**

Regenerate the key using the correct format and re-upload by selecting:

Client Applications > Manage Secrets > Upload Public Key.

1. Regenerate keys in the correct format:

   1. Use PKCS#8 public key (`BEGIN PUBLIC KEY`),
      not SSH-format.
   2. Upload through Client Applications > Manage Secrets.
2. Upload the public key again and re-test. Parsing errors generally
   indicate format issues, not Monitor issues.

## Wrong resource URLs or local host calls

**Symptoms**

- 404 Not Found
- Splunk attempts call to 127.0.0.1:8089 for Docusign
  resources

**Solution**

1. Update Splunk Data Inputs to the correct external endpoints per the
   Add-on.
2. Remove any `localhost/default`  REST manager
   path.
3. Ensure that proper `docusign_accounts`,
   `docusign_users`, and monitor events inputs are
   configured.

## Instance alignment and indexing

**Co-location**

The Splunk App and the Docusign Monitor Add-on must run
on the **same Splunk instance and region**. Mismatched deployment can result in
empty dashboards, even if event ingestion logs confirm data receipt.

**Index Validation**

Confirm the configured target index and sourcetypes (such as
`sourcetype=docusign:monitor)` align with both the data input
settings and dashboard configurations.

## Validate data ingestion

**Initial data ingestion window Retention Policy**

The newly configured Splunk App initiates data retrieval only for the
**last seven days of events**, followed by continuous forward accumulation.

The Monitor Splunk app does not offer Historical backfill. It is not an
automated feature.

**Troubleshooting**

If dashboards display zero values:

- Verify that qualifying events exist within the last seven days.
- Generate a new event to confirm successful ingestion beyond the setup
  phase.

**Validation**

To validate that Docusign Monitor events are polled
through the Splunk Add-on, run this internal audit query:

```
|  |  |
| --- | --- |
|  | index=_internal sourcetype=docusign* sourcetype="docusignmonitor" log_level=* |
|  | "events_received=" |
```

**Diagnostic Scenarios**

Zero Events through Add-on (In Docusign Admin)
:   If events\_received=0 despite the Docusign Admin UI
    reporting activity within the last 7 days:

    - **Data Retention/Sync:** Verify that the polling
      interval is within the seven-day ingestion window.
    - **Trigger test:** Manually generate a new audit
      event in the Docusign Admin UI to force a
      state change.

Architectural Alignment
:   Ensure the Splunk App and Add-on are **co-located** on the same search
    head/instance and mapped to the same region.

Dashboard Nulls (Successful Ingest)
:   If raw data exists but dashboards are vacant, validate **search
    macros** and **lookup tables** (such as
    `docusign_accounts` or
    `docusign_users`). Missing lookups break the schema
    mapping for high-level visualizations.

## User lookup dashboard coverage

**Issue**

Discrepancy between user visibility in the Docusign
Admin and the Splunk User Lookup dashboard.

**Root Cause**

The User Lookup dashboard is dependent on **account-specific input streams**.
Splunk does not automatically discover all accounts under an organization. It only
indexes users for specific Account IDs configured within the Add-on.

**Resolution**

1. **Account IDs:** Specify all relevant Account IDs within the Docusign Organization.
2. **Input Configuration:** Explicitly configure each Account ID as
   an individual input in the Splunk Add-on.
3. **Lookup Refresh:** Re-run the lookup generation
   scripts/searches to populate the user metadata tables for the newly added
   accounts.

## Useful query to check Splunk logs

Confirm that Monitor ingest activity (Add-on internal logs) with this
query:

```
|  |  |
| --- | --- |
|  | index=_internal sourcetype=docusign:modinput:* |
|  | sourcetype="docusign:modinput:monitor" log_level=* events_received" |
```

The query verifies polling and the count of events received on each
call.

## When to escalate a problem

If you completed the troubleshooting steps and ingest still fails after you
corrected entitlement, scopes, keys, versions, and endpoints:

- Collect logs and screenshots and file **a customer support ticket
  at** [**support.docusign.com**](http://support.docusign.com/)**.**
- **Include:**

  - Add-on and App versions
  - Configured indexes and sourcetypes
  - Key provisioning method
  - Recent events timeline (to confirm 7-day window
    behavior).
  - Screenshots and full context
  - Scope listing
  - Splunk platform version
