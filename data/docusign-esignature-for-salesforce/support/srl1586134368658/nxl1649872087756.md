# Use URLFOR Scripting To Build Custom eSignature Buttons

Learn how to use URLFOR scripting to add custom Docusign eSignature
for Salesforce buttons to object layouts.

Important:

URLFOR scripting is a custom solution that is not supported by Docusign
Support. Docusign Professional Services can help you build and
troubleshoot URLFOR custom buttons:

- [Docusign Professional Services
  Overview](https://www.docusign.com/sites/default/files/DocuSign-Professional-Services-Overview.pdf)
- [Packages and services available](https://www.docusign.com/sites/default/files/Enterprise%20Professional%20Services%20Overview.pdf)

You can create many different signing experiences with Docusign eSignature
for Salesforce. You can use a standard Send with Docusign
template, build an envelope template, or build a URLFOR custom button.

You can use URLFOR scripting to build custom signing experiences that aren't provided by the
application. You can add these custom buttons to Salesforce layouts.

URLFOR logic determines what custom actions are performed by the button. The following is a
sample syntax:

{!URLFOR('/apex/dfsle\_\_CustomEnvelopeCreation', null,[sId = Opportunity.Id,OCO =
'Tag'])}

Setting OCO to Tag brings the sender directly to the Prepare
& Send page, where they add Docusign and
Salesforce fields. The Tag field skips the sender past the
Documents and Recipients pages.

You can use the following URLFOR scripting to create customized sending
experiences:

## Custom Contact Note Map

| About this parameter | Technical details |
| --- | --- |
| Parameter Name | CCNM= |
| Values | See Notes |
| Notes | Use this parameter to map a Salesforce Role to a specified Note.  Example: SF Role 1~Note 1;SF Role 2~Note 2;Role 3~Note 3;  The standard selections apply if you do not add a value. |
| Example | {!URLFOR('/apex/dfsle\_\_CustomEnvelopeCreation', null, [sId = Opportunity.Id, CCNM = 'Decision Maker~This Note will be added just to the Decision Maker;Business User~This Note will be added just to the Business User' ] )} |
| Example Description | In this example, Docusign adds “This Note will be added just to the Decision Maker” to the Decision Maker's message.  Docusign adds “This Note will be added just to the Business User” to the personal message of the Business User. |

## Custom Contact Role Map

| About this parameter | Technical details |
| --- | --- |
| Parameter Name | CCRM= |
| Values | See Notes |
| Notes | Use this parameter to map a Salesforce role to a Docusign role. ## Custom Email Message

| About this parameter | Technical details |
| --- | --- |
| Parameter Name | CEM= |
| Values | A text string with the email message. |
| Notes | The message string supports custom tags, such as Opportunity.Name. Example: CEM = 'Please sign this envelope for '+Opportunity.Name  The text string is limited to 2000 characters.  Docusign truncates anything beyond 2000 characters in the email message.  Upgrade to Docusign Apps Launcher 6.4 or higher to use line breaks, commas, and semicolons with the CEM parameter:   - \\n Add a line break to the text - \_COMMA\_ Add a comma to the text - \_SEMI\_ Add a semicolon to the   text   Example scripting:   ``` Hi_COMMA_\\nThis is a wonderful opportunity_SEMI_please sign.\\nThank you. ```   Example email message output:   ``` Hi, This is a wonderful opportunity; please sign. Thank you. ``` |
| Example | {!URLFOR('/apex/dfsle\_\_CustomEnvelopeCreation', null,[sId = Opportunity.Id, CEM = 'Please sign this envelope for '+Opportunity.Name+''])} |
| Example Description | In this example, the customized email message reads, Please sign this envelope for followed by the opportunity name. |

## Custom Recipient List

| About this parameter | Technical details |
| --- | --- |
| Parameter Name | CRL= |
| Values | See Notes |
| Notes | Use this parameter to add recipients to an envelope.  You can add the recipient information manually or dynamically.  Possible values are:   - Email~; - FirstName~; - LastName~; - Role~; - SigningGroup~; - SignInPersonName~; - RoutingOrder~; - AccessCode~; - SMSAuthPhone~; - AccessCode~; - RecipientNote~; - SignNow~, - LoadDefaultContacts~1   You can specify multiple comma-separated recipients. But the length of the entire CRL string must be 1000 characters or fewer.  Example: Docusign adds two recipients from an Opportunity in this scenario:   - Recipient 1: A custom recipient list is created, with one   recipient associated with the opportunity's owner. - Recipient 2: Docusign adds this recipient with   predefined values: CRL =   'Email~'+Opportunity.OwnerEmail+';FirstName~'+Opportunity.OwnerFirstName+';  LastName~'+Opportunity.LastName+';RoutingOrder~1,Email~John.doe@docusign.com;FirstName~John;LastName~Doe;  AccessCode~1234;SMSAuthPhone~5105551212;RecipientNote~Please   sign this document,LoadDefaultContacts~1'   If you do not want to specify both First Name and Last Name, omit the FirstName~ mapping and define the LastName~ mapping.  Your custom object must have a mapping for Email.  Role, RoutingOrder, AccessCode, RecipientNote, SignNow, and SignInPersonName are optional.  RoutingOrder, if specified, must be a positive integer value.  RoutingOrder supersedes any RROS and RRSO options that are specified.  Docusign ignores the RoutingOrder if you add the Docusign Template option.  SignNow, if specified, must be either '0' or  '1'.  SignNow='1': Docusign sends the envelope to the recipient, and the recipient sees the Docusign signing page.  SignNow='0': Docusign sends the envelope to the recipient, and the recipient does not immediately see the Docusign signing page.  SignNow only functions if the recipient's email and user name match the Salesforce user's email and user name.  Docusign ignores SignNow when the recipient's email and user name do not match the Salesforce user's email and user name.  The SignNow recipient must be the first signer when there are multiple signers.  LoadDefaultContacts determines whether the default contacts load for the source object type:   - 0: Do not load - 1: Load   Docusign adds these contacts after the Custom Related Contacts recipients, with the CRL recipient added first.  If you specify DST in the script, the template Routing Order takes precedence over the CRL routing order.  If you specify both CRCL and CRL, Docusign processes CRCL recipients first and then adds CRL recipients.  When creating an in person signer (sign now) recipient with CRL, you must meet the following: requirements:   - The CRL recipient's name and email must match the Salesforce user - The sender's Docusign and Salesforce user names and email addresses   must match - Use the SignNow='1' parameter - The sender must be the first recipient - If you do not set a value for   SignInPersonName, the sender is   the signer   In this scenario, a user sends an envelope and Docusign immediately asks the sender to sign. |
| Example 1 | {!URLFOR('/apex/dfsle\_\_CustomEnvelopeCreation', null, [sId = Lead.Id, CRL='Email~'+Lead.Email+';FirstName~'+Lead.Name+';RoutingOrder~1;Role~Signer 1,SigningGroup~AR;RoutingOrder~2;Role~Signer 2'])} |
| Example 1 Description | In this example, Docusign uses the lead’s name and email for Signer 1. Docusign creates Signer 2 from the AR Signing Group. |
| Example 2 | {!URLFOR('/apex/dfsle\_\_CustomEnvelopeCreation', null, [sId = Account.Id,CRL = 'Email~'+User.Email+';FirstName~'+User.FirstName+';LastName~'+User.LastName+';Role~Sender;SignNow~1'])} |
| Example 2 Description | In this example, Docusign immediately asks the sender to sign. |
