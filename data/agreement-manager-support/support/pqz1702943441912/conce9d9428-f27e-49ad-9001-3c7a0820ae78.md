# Advanced Search Queries

Read about the advanced search query feature that gives you control over how your results are matched and ranked.

Advanced search queries extend your searches beyond basic keywords and filters to control how Agreement Manager finds and ranks your agreements. Use a combination of Boolean operators, exact and proximity searches, boosting, and wildcards in a single query:

![A snippet of the Completed Documents page that highlights an example query string that combines terms, operators, and modifiers.](https://docusign-be-prod.zoominsoftware.io/api/bundle/pqz1702943441912/page/images/Advanced-search-queries_example-querystring_highlighted.png?token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJjdXN0b21lciI6ImRvY3VzaWduX3Byb2R1Y3Rpb24iLCJleHAiOjE3ODE3OTY0MjQsInNoZWFmIjoicHF6MTcwMjk0MzQ0MTkxMiJ9.EF5ob63yt2updb74q_-UDMTEaBXUiAsQB3n2mBLX98M&_LANG=enus)

When you perform a basic keyword search, your query is treated as a loose set of words. Advanced queries give you explicit control to:

- Combine multiple conditions
- Control how closely words appear together
- Emphasize some terms over others

## When to use advanced search queries

Use advanced search queries when you want to:

![An infographic that depicts the reasons for performing advanced search queries.](https://docusign-be-prod.zoominsoftware.io/api/bundle/pqz1702943441912/page/images/Advanced-search-queries_When-do-use-advanced-searches_infographic.png?token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJjdXN0b21lciI6ImRvY3VzaWduX3Byb2R1Y3Rpb24iLCJleHAiOjE3ODE3OTY0MjQsInNoZWFmIjoicHF6MTcwMjk0MzQ0MTkxMiJ9.EF5ob63yt2updb74q_-UDMTEaBXUiAsQB3n2mBLX98M&_LANG=enus)

- Isolate specific clause language within a large document repository. For example, locate
  agreements that mention both a product and a specific risk.
- Exclude noise that cannot be removed with filters. For example, terms that overlap multiple lines of business or teams.
- Find agreements where terms appear near each other, even if the exact wording varies. For example, locate agreements where "landlord" and "indemnify" appear within a few words of each other.
- Search for name or spelling variations without listing every option. For example, you are searching for agreements by party names with varying spellings.
- Combine multiple conditions in a single, reusable query string that can be shared with others or reused later.

## How advanced search queries work

Advanced search queries are entered from the same place in Agreement Manager as basic keyword searches. The difference is how your input is treated. Agreement Manager treats your input as a structured query, not just a list of words. The search looks for special pieces, like operators, quoted phrases, proximity markers, wildcards, and boosts.

![An infographic that depicts how advanced searches work in Agreement Manager.](https://docusign-be-prod.zoominsoftware.io/api/bundle/pqz1702943441912/page/images/Advanced-search-queries_How-advanced-search-works_infographic.png?token=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJjdXN0b21lciI6ImRvY3VzaWduX3Byb2R1Y3Rpb24iLCJleHAiOjE3ODE3OTY0MjQsInNoZWFmIjoicHF6MTcwMjk0MzQ0MTkxMiJ9.EF5ob63yt2updb74q_-UDMTEaBXUiAsQB3n2mBLX98M&_LANG=enus)

The filters that you apply decide *which* agreements are eligible for inclusion. Your advanced query decides:

- *How* the agreement text matches the criteria.
- *Which* matches should rank higher in the results.

When you enter a valid query string, the search returns a focused, predictable result set. The
search engine parses the string into terms, operators, and modifiers, and runs it
against your agreement content.

If the query string is constructed poorly, the search applies filters based on the entered
string. If no relevant filters are detected, a basic keyword search is performed.
You still get search results, but without advanced logic.

## Boolean operators

Use Boolean operators to express must-have, optional, and excluded terms in a single query. Supported operators include: AND, OR, NOT, + (plus), and - (minus).

Review the table to learn more about these operators:

| Operator | Description | Example |
| --- | --- | --- |
| AND | Both terms must be present | msa AND "data center" |
| OR | Either term can be present | "data center" OR "co-location" |
| NOT | The term after NOT must be absent | "data center" NOT "Azure" |
| + (plus) | Term is required (must appear) | +renewal "master agreement" |
| - (minus) | Term must *not* appear | renewal -"evergreen" |

Operators can be combined and grouped with parentheses for more complex logic. For example:

```
|  |  |
| --- | --- |
|  | ("data center" OR "hosting") AND (Azure OR AWS) NOT test |
```

Important: **Rules and restrictions**

- **All caps:** When using Boolean operators, the operators are case-sensitive. Enter the operators in all uppercase (AND, OR, NOT) so they are interpreted as operators, not words.
- **Inside quotes:** Operators that are inside quotes are treated as text, not logic. For example, "AWS and Azure" searches for the phrase literally, not as "AWS" AND "Azure".

## Boosting terms (Boosting)

Boosting is used to favor some terms over others when ranking results, without excluding anything outright. When you enter a term in the search field, add a caret (^) and a weight to indicate its relative importance. For example:

- quick^2 fox (documents matching 'quick' are prioritized or weighted)
- "master services agreement"^3 msa (the exact phrase is prioritized over the abbreviation)

Boosting is helpful when:

- Several terms are important but you want to emphasize one over the others.
- You prefer one term variant over another. For example:

  ```
  |  |  |
  | --- | --- |
  |  | "limitation of liability"^3 "limitation of damages" |
  ```
- Combining broad context with strong signals. For example:

  ```
  |  |  |
  | --- | --- |
  |  | renewal^3 "auto-renew"^2 termination |
  ```

Important: Large boosts can produce unintuitive relevance. Use small integers to start.
The suggested range is 2–5. Boosts work within the advanced query, however,
filters still control which agreements are included in the results. If a boosted
term never appears in the results, it has no effect.

## Exact phrase matching

Exact phrase matching is used to search for words in a specific order, together, rather than anywhere in the agreement. When you know the exact language for which you are searching, enter the phrase in double quotes. For example:

- "Master services agreements"
- "Governing law"
- "Force majeure"

Important: **Rules and restrictions**

- **Order-sensitivity:** Phrases are order-sensitive. For example, "data center" is not the same as "center data".
- **Single search term:** Multi-word quoted phrases are interpreted as a single term in Boolean logic. For example, enter:

  ```
  |  |  |
  | --- | --- |
  |  | msa AND "data center" |
  ```

  to locate master services agreements that contain the quoted phrase, "data center".
- **Multiple quotes, no operators:** If you enter multiple quotes without any operators, the search interprets the string as an AND expression. Both words must be present. For example, enter:

  ```
  |  |  |
  | --- | --- |
  |  | "data" "center" |
  ```

  to locate agreements that contain both words, in any order.

## Proximity search

Proximity searches are used to find phrases where words appear near each other, but not necessarily next to each other. Enter a quoted phrase followed by ~ (tilde) N, where N is the maximum distance (number of words) allowed between terms.

For example, enter:

```
|  |  |
| --- | --- |
|  | "limitation liability"~5 |
```

to search for agreements that
contain both words, within 5 words of each other. The words must appear in the same
agreement, in roughly the same order, within N words of each other.

Proximity searches are helpful when you want to account for:

- **Clause variations:** Find clauses where the wording varies slightly, but the core concept
  stays together.
- **"Near miss" phrases:** Capture phrases while excluding extra words or formatting differences.

Combine your proximity search with Boolean operators to expand the query. For example: enter

```
|  |  |
| --- | --- |
|  | "data center"~5 AND ("AWS" OR "Azure" OR "GCP") |
```

to find agreements that contain "data center" and either AWS, Azure, or GCP. "Data" and "center" can appear within 5 words of each other.

Important: **Rules and restrictions**

- **Large proximity values:** The higher the maximum distance between words, the broader and slower the results. For example, ~50 does not reflect a realistic clause structure. Smaller numbers are recommended.
- **Phrase order:** Proximity searches respect the phrase order. For example, "license lawful"~10 is interpreted differently than "lawful license"~10.
- **Multiple proximity searches:** When you combine multiple proximity searches in one query,
  the complexity is increased. If the performance degrades,
  simplify the query or split it into separate searches.

## Wildcard search

Wildcard searches are used to match word variations without typing every form. Advanced search queries support the use of the \* (asterisk) and ? (question mark) wildcard characters:

| Character | Description | Example |
| --- | --- | --- |
| \* (asterisk) | Matches zero or more characters | - indemnif\* (returns indemnify, indemnification, and indemnified) - bio\* (returns biotech, biological, and biography) |
| ? (question mark) | Matches exactly one character | - organi?ation (returns organization and organisation) - preten?e (returns pretense and pretence) |

Wildcard searches are used to:

- Find name variants
- Handle spelling differences
- Capture families of related terms

Important: **Rules and restrictions**

- **Leading wildcards:** If you enter a wildcard at the beginning of a term, the query scans through a multitude of possibilities. If a query fails or feels slow, move the wildcard to the end or anchor more of the word.
- **Spaces and punctuation:** Wildcards do not match across spaces or punctuation. They typically operate within a single term, as indexed.
- **Combine wildcards for the best results:** Combine wildcards with at least one regular
  keyword or operator instead of using a wildcard-only query.

## Query string search

A query string search is any query that combines multiple operators and features into a single expression. The advanced search uses a query string parser to interpret the full expression that you enter.

The query string parser supports:

- Boolean operators (AND, OR, NOT, +, - )
- Exact phrases and proximity ("...", "..."~N)
- Wildcards (\*, ?)
- Boosting (^N)
- Parentheses for grouping

Full query strings can look like:

```
|  |  |
| --- | --- |
|  | (title:"master services agreement"^3 OR "MSA") AND indemnif* |
```

```
|  |  |
| --- | --- |
|  | ("data center"~5 AND (AWS OR Amazon OR Azure)) NOT test |
```

```
|  |  |
| --- | --- |
|  | +(renewal OR "auto-renew") "notice period"~5 -template |
```

Important: If a query is malformed, the search can return an error and not run the query. Typically, the system reverts to a basic keyword search. You still get results, but without advanced logic. Build your queries incrementally and verify that each piece works on its own. Once verified, combine all the pieces into a single, well-structured query string.

## Recommended practices

Follow these tips to create your advanced search queries:

- **Start simple.** Test a short keyword or phrase first, then layer on
  advanced query operators one at a time.
- **Check operator syntax for Boolean operators.** Use uppercase AND,
  OR, and NOT and keep them outside of any quotes.
- **Watch your quotes and parentheses.** Unbalanced quotes or
  parentheses can cause errors or unexpected results. If your query fails,
  remove the quotes or simplify the groupings and try again.
- **Be careful with wildcard characters.** Avoid leading wildcards and
  broad patterns. They can slow searches or press internal limits. Use
  anchored forms like `cloud*` or
  `indemnif*`.
- **Limit complexity.** Long queries with many query operators are more
  likely to fail or revert to a basic keyword search. If this happens,
  split your logic into smaller queries.
- **Test pieces independently.** If a query stops working, run each part
  on its own to see which component is causing the issue. After isolating
  the issue, fix the component or remove it.
