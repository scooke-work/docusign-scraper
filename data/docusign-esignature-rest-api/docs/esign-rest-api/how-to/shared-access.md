# How to share access to a Docusign envelope inbox

eSignature REST API 2.1 only

This topic demonstrates how to share envelope inbox access with another user and view an envelope using the **SHARED ACCESS** view in the Docusign UI. The Shared Access feature enables a user (the *principal*) to grant another user (the *agent*) access to the principal’s envelopes.

**Important**: There must be envelopes in the principal’s inbox to complete this how-to. See [How to request a signature by email](https://developers.docusign.com/docs/esign-rest-api/how-to/request-signature-email-remote/) to learn more about creating an envelope.

See [Shared Access](https://developers.docusign.com/docs/esign-rest-api/esign101/concepts/envelopes/shared-access/) for details on the feature and the list of supported envelope actions.

Run this code example from our [Quickstart](https://developers.docusign.com/docs/esign-rest-api/quickstart/) or launcher projects: [Bash](https://github.com/docusign/code-examples-bash), [C#](https://github.com/docusign/code-examples-csharp), [Java](https://github.com/docusign/code-examples-java), [Node.js](https://github.com/docusign/code-examples-node), [PHP](https://github.com/docusign/code-examples-php), [PowerShell](https://github.com/docusign/code-examples-powershell), [Python](https://github.com/docusign/code-examples-python), or [Ruby](https://github.com/docusign/code-examples-ruby).

## Required data

Running the code in this how-to requires this data:

| Data element | Description |
| --- | --- |
| {ACTIVATION} | A string value, 50 characters or less, that you provide to the agent to activate their account. This string must not contain the symbols <, >, &, and #. |
| {AGENT\_EMAIL} | A string value for the email address of the agent. |
| {AGENT\_NAME} | A string value for the full name of the agent. |
| {AGENT\_USER\_ID} | A [GUID](http://guid.one/) value that identifies the user. This value is automatically generated by Docusign. Copy the value from the **User ID** field on the [Apps and Keys](https://admindemo.docusign.com/authenticate?goTo=appsAndKeys) page. |
| {API\_ACCOUNT\_ID} | A [GUID](http://guid.one/) value that identifies your account. This value is automatically generated by Docusign for any account you create. Copy the value from the **API Account ID** field on the [Apps and Keys](https://admindemo.docusign.com/authenticate?goTo=appsAndKeys) page. |
| {BASE\_PATH} | A string value that forms the root of the URL required to make API calls. The base path value in the example code is set to target the developer environment at  https://<domain>/restapi  Where <domain> is demo.docusign.net for the developer environment and is the result of the GetUserInfo call in production. |
| {IMPERSONATION\_USER\_GUID} | A [GUID](http://guid.one/) value that identifies the user. This value is automatically generated by Docusign. Copy the value from the **User ID** field on the [Apps and Keys](https://admindemo.docusign.com/authenticate?goTo=appsAndKeys) page. |

## Step 1. Obtain your OAuth token

To make the API calls shown in this how-to, you need a valid OAuth access token. Docusign supports access tokens for three different recommended OAuth grant types: Public Authorization Code Grant, Confidential Authorization Code Grant, and JSON Web Token (JWT) Grant. Choose the OAuth grant type appropriate for your scenario.

See [Confidential Authorization Code Grant](https://developers.docusign.com/platform/auth/authcode/#confidential-authorization-code-grant) for guidance on your options to get an Confidential Authorization Code Grant OAuth token.

## Step 2. Construct your API headers

After you have a valid OAuth access token, you are ready to construct the API headers that you will pass to the API call. Construct your headers like this, but make sure to substitute your valid OAuth access token for {ACCESS\_TOKEN}.

1

2

3

declare -a Headers=('--header' "Authorization:

Bearer ${ACCESS\_TOKEN}"

'--header' "Accept: application/json"

'--header' "Content-Type: application/json")

## Step 3. Construct the request body to create an agent and call the API

To share access to the principal’s envelopes with an agent, there must be at least two users in the principal’s account. This step creates a new agent with whom the principal’s envelope inbox will be shared. If there is already a second user in the principal’s account, you can skip this step and instead record the second user’s user ID to use in the next step.

First, create the request body for the API call to create an agent. Set the `userName` and `email` fields to the name and email address associated with the agent. Set the `activationAccessCode` to a string that the agent will use to activate their account. Make a POST request to the [Users:create](https://developers.docusign.com/docs/esign-rest-api/reference/users/users/create/) endpoint and include the request body to create the agent. Then, extract the agent’s `userId` from the response as {AGENT\_USER\_ID}. You will use it again in Step 4.

## Step 4. Construct the request body to grant authorization and call the API

This step grants shared access to the principal’s Docusign envelope inbox to the agent created in the previous step. If you already have a second user with whom you’d like to share access, you can provide their user ID in the `agentUser` JSON.

Now that you have created an agent, you need to authorize the agent to manage envelopes in the principal’s inbox. Set the `userId` field to the one associated with the agent and set the permission to `manage`. Make a POST request to the Users:authorization endpoint and include the request body to authorize the agent to manage the principal’s envelopes. Use the principal’s user ID as {IMPERSONATION\_USER\_GUID} and the principal’s API account ID as {ACCOUNT\_ID} in the call.

## Expected response

Now that the agent has been authorized, the agent should be able to see the principal’s envelopes in their Shared Access UI. To verify that the principal’s envelopes can be viewed by the agent, log in to the agent’s Docusign developer account, then open the [Agreements tab](https://apps-d.docusign.com/send/documents) in the eSignature UI.  Select **SHARED ACCESS** in the upper left, then select the principal’s account. The principal’s envelopes should be displayed in **Sent**.

## Step 5. Call the eSignature REST API as the agent

The following API call will be made by the agent. Before making this API call, you need to authenticate as the agent.

After you have authenticated as the agent, confirm that the envelope inbox has been shared by calling the [Envelopes:listStatus](https://developers.docusign.com/docs/esign-rest-api/reference/envelopes/envelopes/liststatus/) endpoint. Any API calls made on behalf of the principal need to include the `X-DocuSign-Act-On-Behalf` header and substitute the principal’s user ID for {IMPERSONATION\_USER\_GUID}. Be sure to also substitute the API account ID of the agent for {ACCOUNT\_ID}.

View the source code for this how-to on GitHub: [Bash](https://github.com/docusign/code-examples-bash/blob/master/examples/eSignature/eg043SharedAccess.sh), [C#](https://github.com/docusign/code-examples-csharp/blob/master/launcher-csharp/eSignature/Examples/SharedAccess.cs), [Java](https://github.com/docusign/code-examples-java/blob/master/src/main/java/com/docusign/controller/eSignature/services/SharedAccessService.java), [Node.js](https://github.com/docusign/code-examples-node/blob/master/lib/eSignature/examples/sharedAccess.js), [PHP](https://github.com/docusign/code-examples-php/blob/master/src/Controllers/Examples/eSignature/EG043AuthRequest.php), [PowerShell](https://github.com/docusign/code-examples-powershell/blob/master/examples/eSignature/eg043SharedAccess.ps1), [Python](https://github.com/docusign/code-examples-python/blob/master/app/eSignature/examples/eg043_shared_access.py), or [Ruby](https://github.com/docusign/code-examples-ruby/blob/master/app/services/e_sign/eg043_shared_access_service.rb).
