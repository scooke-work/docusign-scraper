# How to send an envelope to an In Person Signer

This topic demonstrates how to host an [In Person Signing](https://support.docusign.com/s/document-item?rsc_301&bundleId=gav1643676262430&topicId=hak1578456316043.html) session with embedded signing. In Person Signing allows a host to facilitate a Docusign session for a signer who is physically in the same location as them.

Run this code example from our [Quickstart](https://developers.docusign.com/docs/esign-rest-api/quickstart/) or launcher projects: [Bash](https://github.com/docusign/code-examples-bash), [C#](https://github.com/docusign/code-examples-csharp), [Java](https://github.com/docusign/code-examples-java), [Node.js](https://github.com/docusign/code-examples-node), [PHP](https://github.com/docusign/code-examples-php), [PowerShell](https://github.com/docusign/code-examples-powershell), [Python](https://github.com/docusign/code-examples-python), or [Ruby](https://github.com/docusign/code-examples-ruby).

## Required data

Running the code in this how-to requires this data:

| Data element | Description |
| --- | --- |
| {API\_ACCOUNT\_ID} | A [GUID](http://guid.one/) value that identifies your account. This value is automatically generated by Docusign for any account you create. Copy the value from the **API Account ID** field on the [Apps and Keys](https://admindemo.docusign.com/authenticate?goTo=appsAndKeys) page. |
| {BASE\_PATH} | A string value that forms the root of the URL required to make API calls. The base path value in the example code is set to target the developer environment at  https://<domain>/restapi  Where <domain> is demo.docusign.net for the developer environment and is the result of the GetUserInfo call in production. |
| Document to be signed | You must provide at least one document to be signed. Document used in this how-to:  - [PDF file](https://github.com/docusign/code-examples-java/blob/master/src/main/resources/World_Wide_Corp_lorem.pdf) |
| {HOST\_EMAIL} | A string value for the email address associated with the account which will host an In Person Signer. This value must exactly match (case-sensitive) the value shown in the host’s [account profile](https://admindemo.docusign.com/authenticate?goTo=accountProfile). |
| {HOST\_NAME} | A string value for the name associated with the account which will host an In Person Signer. This value must exactly match (case-sensitive) the value shown in the host’s [account profile](https://admindemo.docusign.com/authenticate?goTo=accountProfile). |
| {IN\_PERSON\_SIGNER\_NAME} | A string value for the full name of the signer. |
| {RECIPIENT\_ID} | A unique string value that you assign to identify each recipient. This value assigns tabs on a per-[recipient](https://developers.docusign.com/docs/esign-rest-api/esign101/concepts/recipients/) basis. |
| Return URL | A URL to which the user will be forwarded after signing is complete. |

## Step 1. Obtain your OAuth token

To make the API calls shown in this how-to, you need a valid OAuth access token. Docusign supports access tokens for three different recommended OAuth grant types: Public Authorization Code Grant, Confidential Authorization Code Grant, and JSON Web Token (JWT) Grant. Choose the OAuth grant type appropriate for your scenario.

See [Confidential Authorization Code Grant](https://developers.docusign.com/platform/auth/authcode/#confidential-authorization-code-grant) for guidance on your options to get an Confidential Authorization Code Grant OAuth token.

## Step 2. Construct the envelope definition

Next, create your envelope definition. Add a document to the definition by saving it to a variable as a binary-encoded Base64 string. Then add an [In Person Signer recipient](https://developers.docusign.com/docs/esign-rest-api/reference/envelopes/enveloperecipients/#in-person-signer-recipient) to the envelope. In Person Signing requires that a Docusign account holder act as the host. Set the `hostName` and `hostEmail` fields to the name and email address associated with the host’s Docusign account. Set the `signerName` field to the legal name of the In Person Signer. Then add a `signHereTab` to the recipient and set the envelope’s status to `sent`.

## Step 3. Call the eSignature REST API to create the envelope

Now that you have constructed your API headers and request body, make a POST request to the [Envelopes:create](https://developers.docusign.com/docs/esign-rest-api/reference/envelopes/envelopes/create/) endpoint and include the request body to create the envelope. Save the envelope ID to use later on when creating the recipient view.

## Step 4. Create the recipient view definition

The recipient view is the screen that Docusign generates and displays in your app to enable embedded signing. The accompanying code demonstrates how to write the request body for your recipient view. Set the `returnUrl` to the URL where your users will be sent after signing is complete and use the host’s name and email address for the `userName` and `email` fields. You will not be able to initiate embedded signing if these fields do not have the same value as the `hostName` and `hostEmail` fields from the envelope definition in Step 3.

## Step 5. Call the eSignature REST API to create the recipient view

Next, call the [EnvelopeViews:createRecipient](https://developers.docusign.com/docs/esign-rest-api/reference/envelopes/envelopeviews/createrecipient/) method, passing in your API account ID, envelope ID, and request body generated in previous steps.

Finally, extract the URL for the embedded signing ceremony, which is the value of the `url` key from the JSON response. You can redirect your app's internal browser to this URL to begin signing.

## Expected response

The host will be directed to the screen asking them to host the In-Person Signing session. After choosing **START**, they can hand the device to the in person signer to sign the document. Then the host can complete the signing session per the listed steps.
