# How to set document visibility for envelope recipients

The document visibility feature is available in all developer accounts, but only in certain production account plans. Contact [Docusign Support](https://support.docusign.com/s/contactSupport) or your account manager to find out whether document visibility is available for your production account plan.

This topic demonstrates how to set [document visibility](https://support.docusign.com/s/document-item?bundleId=gbo1643332197980&topicId=eui1578456411411.html) for envelope recipients. Document visibility allows senders to control which documents each recipient can see in an envelope with multiple documents. **Note:** A document cannot be hidden from a recipient if the recipient has tabs assigned to them on the document. Carbon Copy, Certified Delivery (Needs to Sign), Editor, and Agent recipients can always see all documents, unless they are specifically excluded using the `excludedDocuments` property when an envelope is sent. **Important:** You must enable document visibility in your account to complete this how-to:

1. In production, enable the **Allow sender to specify document visibility** parameter in the Sending Settings. This parameter is not displayed in the developer environment Sending Settings, because the feature is enabled on that environment by default.
2. In both production and the developer environment, set the **Document Visibility** parameter in the [Sending Settings](https://admindemo.docusign.com/authenticate?goTo=sending) to any value except **Off**.

See the **Document Visibility** section in [Fields and Properties](https://support.docusign.com/s/document-item?rsc_301&bundleId=pik1583277475390&topicId=xgg1583277350154.html) for details about document visibility options.

## Required data

Running the code in this how-to requires this data:

| Data element | Description |
| --- | --- |
| {API\_ACCOUNT\_ID} | A [GUID](http://guid.one/) value that identifies your account. This value is automatically generated by Docusign for any account you create. Copy the value from the **API Account ID** field on the [Apps and Keys](https://admindemo.docusign.com/authenticate?goTo=appsAndKeys) page. |
| {BASE\_PATH} | A string value that forms the root of the URL required to make API calls. The base path value in the example code is set to target the developer environment at  https://<domain>/restapi  Where <domain> is demo.docusign.net for the developer environment and is the result of the GetUserInfo call in production. |
| {CC\_RECIPIENT\_EMAIL} | A string value for the email address of the carbon copy recipient. |
| {CC\_RECIPIENT\_NAME} | A string value for the name of the carbon copy recipient. |
| Documents to be signed | You must provide at least one document to be signed. Documents used in this how-to:  - [DOCX file](https://github.com/docusign/code-examples-java/blob/master/src/main/resources/World_Wide_Corp_Battle_Plan_Trafalgar.docx) - [PDF file](https://github.com/docusign/code-examples-java/blob/master/src/main/resources/World_Wide_Corp_lorem.pdf) - [HTML file](https://github.com/docusign/code-examples-java/blob/master/src/main/resources/order_form.html) |
| {SIGNER1\_EMAIL} | A string value for the email address where the first signer will receive a notification of the signing request. |
| {SIGNER2\_EMAIL} | A string value for the email address where the second signer will receive a notification of the signing request. |
| {SIGNER1\_NAME} | A string value for the full name of the first signer. |
| {SIGNER2\_NAME} | A string value for the full name of the second signer. |

## Step 1. Obtain your OAuth token

To make the API call shown in this how-to, you need a valid OAuth access token. Docusign supports access tokens for three different recommended OAuth grant types: Public Authorization Code Grant, Confidential Authorization Code Grant, and JSON Web Token (JWT) Grant. Choose the OAuth grant type appropriate for your scenario.

See [Confidential Authorization Code Grant](https://developers.docusign.com/platform/auth/authcode/#confidential-authorization-code-grant) for guidance on your options to get an Confidential Authorization Code Grant OAuth token.

## Step 2. Construct your API headers

After you have a valid OAuth access token, you are ready to construct the API headers that you will pass to the API call. Construct your headers like this, but make sure to substitute your valid OAuth access token for {ACCESS\_TOKEN}.

declare -a Headers=('--header' "Authorization:

Bearer ${ACCESS\_TOKEN}"

'--header' "Accept: application/json"

'--header' "Content-Type: application/json")

## Step 3. Construct the request body

Next, create the request body. To create an envelope using the eSignature REST API, you need to construct an envelope body in JSON format. Replace {CC\_EMAIL}, {CC\_NAME}, {SIGNER1\_EMAIL}, {SIGNER1\_NAME}, {SIGNER2\_EMAIL}, and {SIGNER2\_NAME} with your recipients’ data in the JSON request body.

Include the `enforceSignerVisibility` property set to `true`. This enforces the option selected in the **Document Visibility** section in your account [Sending Settings](https://admindemo.docusign.com/authenticate?goTo=sending) for the envelope. When `enforceSignerVisibility` is set to `true`, you can also override the selected **Document Visibility** option for specific recipients by omitting documents from their envelopes. To omit documents for a recipient, set `excludedDocuments` to an array of the user-defined document IDs that should be hidden from the recipient when they receive the envelope.

## Expected response

- Signer 1 should be able to view and sign “Order acknowledgement”
- Signer 2 should be able to view and sign “Battle Plan Trafalgar” and “Lorem Ipsum”
- Carbon copy recipient should be able to see all three completed documents
