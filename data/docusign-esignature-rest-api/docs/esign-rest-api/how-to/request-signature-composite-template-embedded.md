# How to request a signature using a composite template

This topic demonstrates how to request a signature through your app using a template and an additional, dynamically-constructed HTML document by using the [composite template](https://developers.docusign.com/docs/esign-rest-api/esign101/concepts/templates/composite/) feature.

Run this code example from our [Quickstart](https://developers.docusign.com/docs/esign-rest-api/quickstart/) or launcher projects: [Bash](https://github.com/docusign/code-examples-bash), [C#](https://github.com/docusign/code-examples-csharp), [Java](https://github.com/docusign/code-examples-java), [Node.js](https://github.com/docusign/code-examples-node), [PHP](https://github.com/docusign/code-examples-php), [PowerShell](https://github.com/docusign/code-examples-powershell), [Python](https://github.com/docusign/code-examples-python), or [Ruby](https://github.com/docusign/code-examples-ruby).

## Required data

Running the code in this how-to requires this data:

| Data element | Description |
| --- | --- |
| {API\_ACCOUNT\_ID} | A [GUID](http://guid.one/) value that identifies your account. This value is automatically generated by Docusign for any account you create. Copy the value from the **API Account ID** field on the [Apps and Keys](https://admindemo.docusign.com/authenticate?goTo=appsAndKeys) page. |
| {BASE\_PATH} | A string value that forms the root of the URL required to make API calls. The base path value in the example code is set to target the developer environment at  https://<domain>/restapi  Where <domain> is demo.docusign.net for the developer environment and is the result of the GetUserInfo call in production. |
| {CC\_RECIPIENT\_EMAIL} | A string value for the email address of the carbon copy recipient. |
| {CC\_RECIPIENT\_NAME} | A string value for the name of the carbon copy recipient. |
| {CLIENT\_USER\_ID} | A string value that serves as an identifier of an individual signer throughout the signing process. The combination of this value, the signer's name, and the signer's email is used to uniquely identify them. |
| {ITEM} | An arbitrary string used to specify an item being ordered in the document. |
| {QUANTITY} | An arbitrary string used to specify a quantity of items being ordered in the document. |
| Redirect URI | A URL to which the user will be forwarded after authentication. The value for this URL must exactly match one of those pre-registered for the [integration key](https://developers.docusign.com/platform/configure-app/#integration-key) in your Docusign account. See [redirect URI](https://developers.docusign.com/platform/configure-app/#redirect-uri) for details. |
| {SIGNER\_EMAIL} | A string value for the email address where the signer will receive a notification of the signing request. |
| {SIGNER\_NAME} | A string value for the full name of the signer. |
| {TEMPLATE\_ID} | A [GUID](http://guid.one/) value that identifies a Docusign [template](https://developers.docusign.com/docs/esign-rest-api/esign101/concepts/templates/) that is used to create a Docusign [envelope](https://developers.docusign.com/docs/esign-rest-api/esign101/concepts/envelopes/). This value is automatically generated by Docusign when the template is created. |

## Step 1. Obtain your OAuth token

To make the API calls shown in this how-to, you need a valid OAuth access token. Docusign supports access tokens for three different recommended OAuth grant types: Public Authorization Code Grant, Confidential Authorization Code Grant, and JSON Web Token (JWT) Grant. Choose the OAuth grant type appropriate for your scenario.

See [Confidential Authorization Code Grant](https://developers.docusign.com/platform/auth/authcode/#confidential-authorization-code-grant) for guidance on your options to get an Confidential Authorization Code Grant OAuth token.

## Step 2. Create the envelope definition as a part of the composite template

First, create a new envelope definition that declares a pair of `compositeTemplate` objects. Each of these objects will represent one template that you will use to derive the created envelope. The first of these composite templates will be derived from one of your existing templates, specified by its template ID. It is divided into a `serverTemplate` (which holds server-side template data, such as ordering) and an `inlineTemplate` (which holds inline template data, such as tabs/fields). The second composite template will contain the HTML document, as well as its tabs and recipients.

**Note:** When you create an envelope by using a composite template, you should specify the envelope custom fields in the inline template. Any custom fields that you specify at the root level are ignored.

## Step 3. Call Docusign to create the envelope

Next, create the envelope from the definition that you created in the previous step, then create the recipient view for embedded signing.

1. Instantiate the Docusign API client.
2. Set the following default request header to add your access token.
   `"Authorization" : "Bearer {ACCESS_TOKEN}"`
3. Initialize the Envelopes API.
4. Create an envelope by passing in the generated envelope definition and your API account ID.

--request POST ${base\_path}/v2.1/accounts/${account\_id}/

envelopes \

--output $response

## Step 4. Create the recipient view definition

The recipient view is the screen that Docusign generates and displays in your app to enable your signers to sign documents. The recipient view request returns a link that enables your recipients to sign documents from your app. To create the definition, you must add the signer's information and the redirect URI.

1. Set the redirect URI to which the user will be forwarded after signing is complete. This URI must exactly match one that is preregistered for the integration key in your Docusign account.

   **Note**: You can include a `state` parameter in the URI string to record your application's state and provide recovery information, potentially enabling it to resume interrupted operations using only callback data.
2. Add the recipient information to the view, including the signer's email, name, and client user ID.
