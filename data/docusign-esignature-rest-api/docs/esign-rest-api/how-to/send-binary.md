# How to attach documents via binary transfer

This topic demonstrates how to attach documents to an envelope via binary transfer.

Run this code example from our [Quickstart](https://developers.docusign.com/docs/esign-rest-api/quickstart/) or launcher projects: [Bash](https://github.com/docusign/code-examples-bash), [C#](https://github.com/docusign/code-examples-csharp), [Java](https://github.com/docusign/code-examples-java), [Node.js](https://github.com/docusign/code-examples-node), [PHP](https://github.com/docusign/code-examples-php), [PowerShell](https://github.com/docusign/code-examples-powershell), [Python](https://github.com/docusign/code-examples-python), or [Ruby](https://github.com/docusign/code-examples-ruby).

## Required data

Running the code in this how-to requires this data:

| Data element | Description |
| --- | --- |
| {API\_ACCOUNT\_ID} | A [GUID](http://guid.one/) value that identifies your account. This value is automatically generated by Docusign for any account you create. Copy the value from the **API Account ID** field on the [Apps and Keys](https://admindemo.docusign.com/authenticate?goTo=appsAndKeys) page. |
| {BASE\_PATH} | A string value that forms the root of the URL required to make API calls. The base path value in the example code is set to target the developer environment at  https://<domain>/restapi  Where <domain> is demo.docusign.net for the developer environment and is the result of the GetUserInfo call in production. |
| {BOUNDARY\_STRING} | A string value that acts as a delimiter for multipart body data. This value should not be found in any documents or fields. |
| {CC\_RECIPIENT\_EMAIL} | A string value for the email address of the carbon copy recipient. |
| {CC\_RECIPIENT\_NAME} | A string value for the name of the carbon copy recipient. |
| {CLIENT\_USER\_ID} | A string value that serves as an identifier of an individual signer throughout the signing process. The combination of this value, the signer's name, and the signer's email is used to uniquely identify them. |
| {DOCS\_PATH} | A string value for the file path to the folder containing the documents you want to include in the [envelope](https://developers.docusign.com/docs/esign-rest-api/esign101/concepts/envelopes/). |
| Documents to be signed | You must provide at least one document to be signed. Documents used in this how-to:  - [DOCX file](https://github.com/docusign/code-examples-java/blob/master/src/main/resources/World_Wide_Corp_Battle_Plan_Trafalgar.docx) - [PDF file](https://github.com/docusign/code-examples-java/blob/master/src/main/resources/World_Wide_Corp_lorem.pdf) - A third document in HTML is constructed dynamically by the sample code |
| {SIGNER\_EMAIL} | A string value for the email address where the signer will receive a notification of the signing request. |
| {SIGNER\_NAME} | A string value for the full name of the signer. |

## Step 1. Obtain your OAuth token

To make the API call shown in this how-to, you need a valid OAuth access token. Docusign supports access tokens for three different recommended OAuth grant types: Public Authorization Code Grant, Confidential Authorization Code Grant, and JSON Web Token (JWT) Grant. Choose the OAuth grant type appropriate for your scenario.

See [Confidential Authorization Code Grant](https://developers.docusign.com/platform/auth/authcode/#confidential-authorization-code-grant) for guidance on your options to get an Confidential Authorization Code Grant OAuth token.

## Step 3. Construct the request body

Create a custom envelope definition to apply multipart boundary string delimiters for your JSON request. This topic shows how to create a JSON envelope definition using a method, `makeEnvelope`, that:

- Creates your multipart JSON request body object.
  - Begin a multipart body by first appending two hyphens, then the boundary separator. From there, append a CRLF (A carriage return/line feed, more commonly known as a line break or new line), then add the header `"Content-Type: application/json"`; make another CRLF, and then add the header `"Content-Disposition: form data"`.
  - Now add two more CRLFs, one after another (which occurs to denote that the proceeding data is assumed to be of `Content-Type text/plain`). From there, insert the envelope JSON object.
- Attaches multiple documents to the JSON envelope.
  - Add each file in bytes (specifically as an encoded byte array), with a correlated content disposition, into the request body. To do so, for each document, append your existing request body with the following sequence: `Content-Type` with the value being the MIME type, UTF-8 encoded, and then `Content-Disposition` as a file, also including attributes such as the filename and document ID, and finally the document in bytes.
- Selects the appropriate MIME type and file attributes for your documents.
  > **Note:** You must specify correct matching MIME-type attributes so you do not encounter errors when attempting to load the documents.
- Specifies your CRLF boundary delimiter and applies it to your documents.
  - With all documents attached one after another, split by boundary delimiters, specify a closing boundary by appending a final set of CRLF, hyphens, boundary, hyphens, then ending CRLF.
- Attaches multiple documents to the JSON envelope.
- Sets signer and cc recipient data in the JSON envelope.
- Creates a routing order for the documents in the JSON envelope. Documents are delivered to recipients in ascending order, as each previous document is signed and completed.
- Creates and assigns signable fields to specific signers within the documents.

## Step 4. Call the eSignature REST API

With a completed request body taking some reference fields from the envelope JSON, you can send the envelope with the [Envelopes:create](https://developers.docusign.com/docs/esign-rest-api/reference/envelopes/envelopes/create/) method.

- Send a POST request using the base path that injects your API account ID in the `account_id` portion of the URI.
- Set the `Authorization` header to the OAuth2.0 bearer token, an `Accept` header to `application/json`, the `Content-type` to `multipart/form-data`, and the boundary to the specified delimiter variable in the prerequisite. Complete the request by passing in the completed request body.
- Keep the code contained in a wrapper named `worker`, then run this worker method within a `try/catch` mechanism to handle any returned errors should any parameters be invalid or expired. By using a known template GUID and passing along appropriate template role parameters, you can return an envelope ID from the result.

View the source code for this how-to on GitHub: [Bash](https://github.com/docusign/code-examples-bash/blob/master/examples/eSignature/eg010SendBinaryDocs.sh), [C#](https://github.com/docusign/code-examples-csharp/blob/master/launcher-csharp/eSignature/Examples/CreateEnvelopeWithMultipleDocumentTypes.cs), [Java](https://github.com/docusign/code-examples-java/blob/master/src/main/java/com/docusign/controller/eSignature/examples/EG010ControllerSendBinaryDocs.java), [Node.js](https://github.com/docusign/code-examples-node/blob/master/lib/eSignature/examples/sendBinaryDocs.js), [PHP](https://github.com/docusign/code-examples-php/blob/master/src/Services/Examples/eSignature/SendBinaryDocsService.php), [PowerShell](https://github.com/docusign/code-examples-powershell/blob/master/examples/eSignature/eg010SendBinaryDocs.ps1), [Python](https://github.com/docusign/code-examples-python/blob/master/app/eSignature/examples/eg010_send_binary_docs.py), or [Ruby](https://github.com/docusign/code-examples-ruby/blob/main/app/services/e_sign/eg010_send_binary_docs_service.rb).
