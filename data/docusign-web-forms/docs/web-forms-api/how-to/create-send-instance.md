# How to create and send a web form instance

The Web Forms API is available in all developer accounts, but only in certain production account plans. Contact [Docusign Support](https://support.docusign.com/s/?language=en_US) or your account manager to find out whether the Web Forms API is available for your production account plan.

This topic demonstrates how to create a web form instance and send it to a recipient by email ([remote web form instance delivery](https://developers.docusign.com/docs/web-forms-api/plan-integration/deliver-remote-instances-to-users/)). After the recipient fills out the form, they will sign an envelope.

Run this code example from our [Quickstart](https://developers.docusign.com/docs/esign-rest-api/quickstart/) or launcher projects: [Bash](https://github.com/docusign/code-examples-bash), [C#](https://github.com/docusign/code-examples-csharp), [Java](https://github.com/docusign/code-examples-java), [Node.js](https://github.com/docusign/code-examples-node), [PHP](https://github.com/docusign/code-examples-php), [PowerShell](https://github.com/docusign/code-examples-powershell), [Python](https://github.com/docusign/code-examples-python), or [Ruby](https://github.com/docusign/code-examples-ruby).

## Prerequisites

Web form configurations cannot be created through the API. Therefore, before you create a web form instance, you must [create](https://support.docusign.com/s/document-item?bundleId=gmi1660583110357&topicId=euk1665093155272.html) and [activate](https://support.docusign.com/s/document-item?bundleId=gmi1660583110357&topicId=iri1669070601520.html) a web form configuration in your Docusign account. If you are running this from our Quickstart or launcher projects, you will receive instructions for how to create a web form configuration by uploading a compatible JSON configuration file.

If you choose not to use our launcher projects, you can follow along with this guide by creating a web form configuration with the name “Web Form Example Template” and adding the following fields:

- A text field with the API reference name “PhoneNumber”
- A text field with the API reference name “Company”
- A text field with the API reference name “JobTitle”
- A checkbox with the API reference name “Yes”

## Required data

Running the code in this how-to requires this data:

| Data element | Description |
| --- | --- |
| {API\_ACCOUNT\_ID} | A [GUID](http://guid.one/) value that identifies your account. This value is automatically generated by Docusign for any account you create. Copy the value from the **API Account ID** field on the [Apps and Keys](https://admindemo.docusign.com/authenticate?goTo=appsAndKeys) page. |
| {BASE\_PATH} | A string value that forms the root of the URL required to make API calls. The base path value in the example code is set to target the developer environment at   https://<domain>/webforms/v1.1  Where <domain> is apps-d.docusign.com/api for the developer environment. |

## Step 1. Obtain your OAuth Token

To make the API calls shown in this how-to, you need a valid OAuth access token. Docusign supports access tokens for three different recommended OAuth grant types: Public Authorization Code Grant, Confidential Authorization Code Grant, and JSON Web Token (JWT) Grant. Choose the OAuth grant type appropriate for your scenario.

Web Forms OAuth access tokens are the same ones used for our [eSignature REST API](https://developers.docusign.com/docs/esign-rest-api/esign101/auth/), although you must request one or more Web Forms scopes to call the Web Forms API. The API calls made in this how-to require that you request the eSignature REST API `signature` scope and Docusign Web Forms–specific scopes `webforms_instance_read`, `webforms_instance_write`, and `webforms_read` when obtaining your access token.

See [Authentication](https://developers.docusign.com/docs/web-forms-api/plan-integration/authentication/) for guidance on your options to get an OAuth token.

## Step 2. Construct your API headers

After you have a valid OAuth access token, you are ready to construct the API headers that you will pass to the API call. Construct your headers like this, but make sure to substitute your valid OAuth access token for {ACCESS\_TOKEN}.

declare -a Headers=('--header' "Authorization:

Bearer ${ACCESS\_TOKEN}" \

'--header' "Accept: application/json" \

'--header' "Content-Type: application/

json")

## Step 3. Get the form ID

First, get the ID of the web form configuration of which you will be creating an instance. Call the [Configurations:listForms](https://developers.docusign.com/docs/web-forms-api/reference/webforms/configurations/listforms/) endpoint, passing the value `Web Form Example Template` to the `search` query parameter. This query parameter filters the results of the API call so that only web form configurations with the name “Web Form Example Template” are returned. Save the value of `id` from the response so that it can be used in Step 5.

response=$(mktemp /tmp/response-cw.XXXXXX)

FORM\_ID=$(cat $response | sed 's/,/\n/g' | grep id |

sed 's/.\*\"id\":\"//g' | sed 's/\".\*//g')

## Step 4. Construct the request body

Next, create the request body. 

Use the `formValues` object to [prefill](https://developers.docusign.com/docs/web-forms-api/plan-integration/prefill-instance-fields/) the fields in the web form instance. The keys in this object correspond to the API reference names of the fields in the web form configuration. You can find and edit these API reference names in the [Properties tab](https://support.docusign.com/s/document-item?language=en_US&bundleId=gmi1660583110357&topicId=wor1676501633875.html) in the web forms UI.

When `sendOption` is set to `now`, the request must supply required information for each recipient. In this example, the information is supplied in a `recipients` array, where each `recipient` object contains a `roleName`, `name`, and `email`. As an alternative, required user information can be supplied in the `formValues` object. See [Deliver remote web form instances to users](https://developers.docusign.com/docs/web-forms-api/plan-integration/deliver-remote-instances-to-users/) for details about this option.

## Step 5. Call the Web Forms API

Now that you have constructed your API headers and request body, make a POST request to the [Instances:createInstance](https://developers.docusign.com/docs/web-forms-api/reference/webforms/instances/createinstance/) method and include the request body to create a new instance of the web form.

Here’s how you make the call, but make sure to substitute your API account ID for {ACCOUNT\_ID} and form ID for {FORM\_ID}.

View the source code for this how-to on GitHub: [Bash](https://github.com/docusign/code-examples-bash/blob/master/examples/WebForms/eg002CreateRemoteInstance.sh), [C#](https://github.com/docusign/code-examples-csharp/blob/master/launcher-csharp/WebForms/Examples/CreateRemoteInstanceService.cs), [Java](https://github.com/docusign/code-examples-java/blob/master/src/main/java/com/docusign/controller/webForms/services/CreateRemoteInstanceService.java), [Node.js](https://github.com/docusign/code-examples-node/blob/master/lib/webforms/examples/createRemoteInstance.js), [PHP](https://github.com/docusign/code-examples-php/blob/master/src/Services/Examples/WebForms/CreateRemoteInstanceService.php), [PowerShell](https://github.com/docusign/code-examples-powershell/blob/master/examples/WebForms/eg002CreateRemoteInstance.ps1), [Python](https://github.com/docusign/code-examples-python/blob/master/app/webforms/examples/eg002_create_remote_instance.py), or [Ruby](https://github.com/docusign/code-examples-ruby/blob/master/app/services/webforms/eg002_create_remote_instance_service.rb).

## Expected response

If you run this code example from our Quickstart or launcher projects, you should see a JSON response similar to the code shown.

In addition, the signer receives an envelope email containing a link. On selecting the link, the user is routed to the Docusign platform, which displays an envelope with an embedded web form instance. The user fills out the web form instance and then electronically signs an envelope containing fields prefilled with the information entered in the web form. After the signing process is finished, an email is sent to the recipient, containing a copy of the signed envelope.
